# Module 16: Login UI (หน้าเข้าสู่ระบบ)

> ไฟล์นี้เป็นรายละเอียดเฉพาะของ Module 16 อ่านไฟล์นี้เมื่อเริ่มทำงาน module นี้เท่านั้น
> ต่อยอดจาก Module 9 (Auth logic ฝั่ง backend) และ Module 10 (User & Role)
> อ้างอิงบริบทรวมจาก `CLAUDE.md` และ skill `ui-design-system`

## เป้าหมายของ Module

Module 9 สร้าง auth logic ฝั่ง backend ไว้แล้ว (login/logout) แต่ยังไม่มี module ไหนออกแบบ**หน้าจอ login ที่สมบูรณ์และตรงกับ design system เดียวกับหน้าอื่นในระบบ** — mockup เดิมที่ทำไว้ตอน pitch ข้ามตรงไปที่หน้า dashboard หลัง login แล้วเลย

Module นี้เพิ่มหน้าจอที่ผู้ใช้จะเห็นเป็นด่านแรกสุดของระบบ ซึ่งสำคัญเพราะเป็นจุดแรกที่ผู้สอน/ผู้ใช้จริงจะเห็นตอนเปิดเว็บแอปขึ้นมา

## หน้าที่ต้องสร้าง

| หน้า | เส้นทาง (route) | เข้าถึงได้เมื่อไร |
|---|---|---|
| หน้าเข้าสู่ระบบ (Login) | `/login` | ยังไม่ได้ login เท่านั้น (ถ้า login แล้วเข้าหน้านี้ ให้ redirect ไป `/dashboard` อัตโนมัติ) |
| หน้าลืมรหัสผ่าน (Forgot Password) | `/forgot-password` | ยังไม่ได้ login |
| หน้าตั้งรหัสผ่านใหม่ (Reset Password) | `/reset-password` | เข้าถึงผ่านลิงก์ที่ส่งไปทางอีเมลเท่านั้น (มี token ตรวจสอบ) |

> ระบบนี้เป็นระบบภายในหน่วยงาน (ไม่ใช่สาธารณะ) จึง**ไม่ต้องมีหน้า "สมัครสมาชิก" (Register) แบบเปิดให้ใครก็สมัครได้** — การสร้างบัญชีผู้ใช้ทำผ่านหน้า "จัดการผู้ใช้งาน" โดย Admin เท่านั้น (ตาม Module 10) ถ้า Claude เข้าใจผิดแล้วเริ่มสร้างหน้า register แบบสาธารณะ ให้หยุดและแก้ไขทันที

## องค์ประกอบของหน้า Login

จัด layout กึ่งกลางหน้าจอ (ไม่ใช้ sidebar เพราะยังไม่ login):

1. **โลโก้/ชื่อระบบ** ด้านบน — ใช้ตราสัญลักษณ์เดียวกับที่อยู่บน sidebar ของหน้าอื่น (ตาม skill `ui-design-system`)
2. **ฟอร์ม login**
   - ช่องกรอกอีเมล/ชื่อผู้ใช้
   - ช่องกรอกรหัสผ่าน (มีปุ่มแสดง/ซ่อนรหัสผ่าน)
   - Checkbox "จดจำการเข้าสู่ระบบ" (ถ้าต้องการ)
   - ปุ่ม "เข้าสู่ระบบ" (ใช้สี primary ตาม design system)
   - ลิงก์ "ลืมรหัสผ่าน?" ใต้ฟอร์ม
3. **ข้อความ error** แสดงใต้ฟอร์มเมื่อ login ไม่สำเร็จ (อีเมล/รหัสผ่านผิด, บัญชีถูกปิดใช้งาน) — ข้อความต้องไม่บอกรายละเอียดเกินจำเป็น เช่น ใช้ "อีเมลหรือรหัสผ่านไม่ถูกต้อง" แบบเดียวกันทั้งกรณีอีเมลไม่มีในระบบและกรณีรหัสผ่านผิด (ป้องกันการเดาว่าอีเมลไหนมีอยู่ในระบบจริง)
4. **Loading state** ขณะกำลังตรวจสอบข้อมูล login (ปุ่มแสดง spinner ระหว่างรอ)

## การเชื่อมกับ Module อื่น

- **Module 9 (Auth logic):** หน้านี้เรียกใช้ API/logic ที่ Module 9 สร้างไว้แล้วโดยตรง ไม่ต้องเขียน authentication logic ใหม่
- **Module 10 (User & Role):** หลัง login สำเร็จ redirect ไปหน้าที่เหมาะกับ role ของผู้ใช้ (ทุก role ไปหน้า dashboard เดียวกันได้ เพราะ sidebar จะซ่อน/แสดงเมนูตาม role อยู่แล้ว)
- **Module 12 (Audit Log):** ทุกครั้งที่ login สำเร็จ/ไม่สำเร็จ ต้องเรียก `logAction()` บันทึก `USER_LOGIN` หรือ `USER_LOGIN_FAILED` ตามที่ระบุไว้ใน module-12-audit-log.md อยู่แล้ว — ตรวจสอบว่าหน้า login ใหม่นี้เชื่อมกับ log ถูกต้อง ไม่ใช่แค่ backend endpoint เดิมที่อาจไม่ได้ผ่านหน้า UI นี้

## Session/Route Protection (สำคัญ)

- สร้าง middleware กลาง (`src/middleware.ts` ของ Next.js) ตรวจสอบทุก route ที่ต้อง login ก่อนเข้าถึง — ถ้ายังไม่ login ให้ redirect ไป `/login` อัตโนมัติ พร้อมจดจำ path เดิมไว้ (`?redirect=/documents/create`) เพื่อพาผู้ใช้กลับไปหน้าที่ตั้งใจจะเข้าหลัง login สำเร็จ
- หน้า `/login`, `/forgot-password`, `/reset-password` เป็น**หน้าเดียวที่เข้าถึงได้โดยไม่ต้อง login** ทุกหน้าอื่นต้องผ่าน middleware นี้เสมอ

## Testing Checklist

- [ ] Login ด้วยข้อมูลถูกต้อง → เข้าสู่ dashboard สำเร็จ
- [ ] Login ด้วยอีเมล/รหัสผ่านผิด → แสดง error message เดียวกันทั้งสองกรณี (ไม่บอกว่าอีเมลถูกหรือรหัสผ่านผิด)
- [ ] Login ด้วยบัญชีที่ถูกปิดใช้งาน (`isActive = false` จาก Module 10) → ปฏิเสธการเข้าสู่ระบบ พร้อมข้อความที่เหมาะสม
- [ ] เข้า URL ของหน้าที่ต้อง login โดยตรง (เช่น `/dashboard`) ตอนยังไม่ได้ login → redirect ไป `/login` อัตโนมัติ
- [ ] Login สำเร็จแล้วเข้าหน้า `/login` ซ้ำ → redirect ไป `/dashboard` อัตโนมัติ ไม่แสดงฟอร์ม login ซ้ำ
- [ ] ทุกความพยายาม login (สำเร็จ/ไม่สำเร็จ) ถูกบันทึกใน Audit Log ตรงตาม Module 12
