## 5. ปัญหา Git ที่เจอและวิธีแก้ (สำคัญมาก — ใช้อ้างอิงครั้งต่อไป)

| ปัญหา | สาเหตุ | วิธีแก้ |
|---|---|---|
| Skill บางตัวไม่ปรากฏบนเครื่อง server หลัง pull | ไฟล์ถูก commit ไปอยู่บน feature branch (เช่น `feature/upgrade-ui`) ไม่ใช่ `master` | `git checkout <feature-branch> -- <path>` ดึงเฉพาะไฟล์มาที่ `master` แล้ว commit ใหม่ |
| `git checkout main` ขึ้น error `pathspec 'main' did not match` | repo นี้ใช้ชื่อ branch หลักว่า `master` ไม่ใช่ `main` | ใช้ `master` แทนทุกจุด |
| `git push` ขึ้น `[rejected] (fetch first)` | remote มี commit ที่ local ไม่มี (มีการ push จากอีกเครื่อง/อีกช่วงเวลา) | `git pull origin master` ก่อน แล้วค่อย push |
| `git pull` ขึ้น `divergent branches, need to specify how to reconcile` | local และ remote มีประวัติคนละชุดพร้อมกัน | `git config pull.rebase false` (เลือก merge ไม่ใช่ rebase) แล้ว `git pull` ใหม่ หรือใช้ `git pull --no-rebase origin master` ตรงๆ |
| commit ที่มีข้อความว่าใส่ 2 ไฟล์ แต่จริงมีแค่ 1 ไฟล์ | คำสั่ง `git checkout <branch> -- <path>` ก่อนหน้าล้มเหลวเงียบๆ (path ผิด) แต่ commit ทั้งสองคำสั่งไปในครั้งเดียว | ตรวจสอบด้วย `git log --oneline -- <path>` เทียบทีละไฟล์เสมอ อย่าเชื่อแค่ข้อความ commit |

**บทเรียนรวม:** เอกสาร/skill ควร commit แยกออกมาที่ `master` โดยตรงเสมอ ไม่ควรพ่วงไปกับ feature branch ที่กำลังพัฒนาโค้ดอยู่

## 6. Auth Bug ที่แก้ไปแล้ว

**อาการ:** หลัง login สำเร็จ กด Logout แล้ว redirect ไปที่ `http://localhost:3000` เกิด `ERR_CONNECTION_REFUSED` ทั้งที่เข้าเว็บผ่าน IP วง LAN (`192.168.1.152`)

**สาเหตุ:** Auth.js (next-auth v5) ตั้งค่า `AUTH_URL` แบบ hardcode เป็น `localhost:3000` ทำให้ระบบสร้าง redirect link ผิดเมื่อเข้าผ่าน IP อื่น

**วิธีแก้ที่เลือกใช้:** ตั้ง `AUTH_TRUST_HOST=true` แทนการ hardcode `AUTH_URL` — ให้ Auth.js อ่าน host จาก request จริงแทน (ยืดหยุ่นกว่า รองรับการเปลี่ยน IP/ใส่ Nginx ในอนาคตโดยไม่ต้องแก้ค่านี้ซ้ำ)

- มี 5 commits ค้างอยู่บนเครื่อง server ที่ยังไม่ได้ push (รวม 2 security fix สำคัญ)
- `docker` group ของ session บนเครื่อง server ยังไม่ refresh (ต้อง logout/login)
- `docs/progress-log.md` ล้าสมัย ต้องอัปเดตให้ตรงกับ commit ล่าสุดจริง

### ขั้นตอนที่วางแผนไว้ต่อจากนี้ (ยังไม่ได้ทำ ณ จุดจบบทสนทนา)
1. Logout/login เพื่อ refresh docker group
2. Push 5 commits ค้างขึ้น origin
3. สร้าง `.env.production` และ `.env.test` (secret สุ่มใหม่ทั้งหมด ห้าม copy จาก `.env`)
4. รัน `prisma migrate deploy` กับ production database
5. อัปเดต `docs/progress-log.md`
6. รัน Testing (Module 14) + Security Review (skill `security-review`, แนะนำใช้ Opus)
7. ทดสอบ Backup/Restore จริงบนเครื่อง (สำคัญที่สุดเพราะไม่มี Snapshot)
8. Smoke test ผ่านหน้าเว็บจริงจาก LAN
9. ล้างข้อมูลทดสอบ + สร้างบัญชีผู้ใช้จริง
10. Tag เวอร์ชัน (`v1.0.0-golive`) และปิดจบ

### 10.3 "สัดส่วนแยกตามประเภทเอกสาร" → bar chart คลิกได้
- โจทย์: ทำให้กราฟสวยขึ้น + คลิกแท่งแล้วพาไปหน้าเอกสารทั้งหมดกรองตามประเภทนั้น
- **ลองใช้ `recharts` (ผ่าน `npx shadcn add chart`) ก่อน แล้วเจอบั๊กจริงใน recharts 3.8.0:** `<BarChart layout="vertical">` คำนวณ `width` ของทุกแท่งเป็น `0` เสมอไม่ว่าค่าจริงจะเท่าไหร่ (ไล่เข้าไปดู source ถึงระดับ `computeBarRectangles`/`Rectangle` component พบว่า `xAxis.scale.map(value)` คืนค่าตำแหน่งเดียวกับ baseValue ทั้งที่ domain/ticks ที่ render จริงถูกต้อง — เป็นบั๊กภายในเวอร์ชันนี้ ไม่ใช่การตั้งค่าผิด)
- **ตัดสินใจ:** ถอด `recharts` และ `src/components/ui/chart.tsx` (shadcn wrapper) ออกจากโปรเจกต์ทั้งหมด แล้วเขียน bar chart เองด้วย Tailwind ล้วน (`<Link>` + สอง `<div>` ซ้อนกันเป็น track/fill) — ง่ายกว่า ไม่มีความเสี่ยงจากบั๊ก library และ dependency footprint เล็กกว่าสำหรับกราฟง่ายๆ แค่จุดเดียวในระบบ
- **ห้ามติดตั้ง `recharts` ใหม่โดยไม่ตรวจสอบบั๊กนี้ก่อน** (ดูรายละเอียดใน `.claude/skills/ui-design-system/SKILL.md` หัวข้อ "Dashboard Breakdown Chart")
- Bar เทียบสัดส่วนกับค่าสูงสุดในชุดข้อมูล (ไม่ใช่ผลรวม) เพื่อให้แท่งที่มากที่สุดเต็มความกว้างเสมอ — อ่านง่ายกว่าเดิม
- ต้องส่ง `documentTypeCode` เพิ่มจาก `getDashboardStats()` (เดิมมีแค่ `documentTypeId`) และแก้หน้า `/documents` ให้อ่าน query param `?type=` มา pre-filter ให้ `DocumentsTable` ตั้งแต่โหลดหน้า

### 11.4 ฟอร์ม "เพิ่ม/แก้ไขผู้ใช้งาน" — เพิ่มรายละเอียดให้ครบขึ้น
- โจทย์: แยกชื่อ-นามสกุล, เพิ่ม dropdown คำนำหน้าชื่อ (นาย/นาง/นางสาว) และ dropdown ฝ่าย (บริหาร/ทรัพยากรบุคคล/การเงินและบัญชี/การตลาด/ขาย/ไอที/บริการลูกค้า)
- **ก่อนแก้ไขได้ยืนยันกับผู้ใช้ 2 จุดสำคัญก่อนเริ่มโค้ด** (ผู้ใช้สั่งห้ามลงมือจนกว่าจะอนุมัติ):
  1. **`division` (ฝ่าย) ต้องเป็นคนละฟิลด์กับ `departmentCode` (หน่วยงาน) เดิมโดยเด็ดขาด** — `departmentCode` คือรหัสย่อ 2-4 ตัวอักษรที่ระบบเลขที่เอกสารอัตโนมัติใช้เป็น prefix (`ศรพ`, `ทสบ`) ส่วนฝ่ายที่ผู้ใช้ขอ (ฝ่ายการตลาด, ฝ่ายขาย ฯลฯ) เป็นชื่อเต็มแบบองค์กรทั่วไป คนละความหมายกันโดยสิ้นเชิง ถ้าเผลอรวมกันจะพังระบบออกเลขที่เอกสาร
  2. **เพิ่มฟิลด์ `position` (ตำแหน่งงาน)** ตามคำแนะนำ เพราะเอกสารมีช่อง "ตำแหน่งผู้ลงนาม" (`signerPosition`) ที่ต้องพิมพ์เองทุกครั้งอยู่แล้ว การมี field นี้ใน User profile จะช่วยเติมอัตโนมัติได้ในอนาคต
- Schema: เพิ่ม `titlePrefix`/`firstName`/`lastName`/`division`/`position` แบบ nullable ทั้งหมด (ไม่กระทบบัญชีที่มีอยู่ก่อน) เก็บ `name` เป็นชื่อเต็มคำนวณอัตโนมัติต่อไปเพื่อไม่ให้จุดอื่นในระบบที่อ้าง `user.name` พัง
- **บั๊กที่เจอระหว่างทดสอบเอง (ไม่เกี่ยวกับโจทย์ แต่บล็อกการทดสอบ เลยแก้ไปด้วย):**
  1. field optional แบบ text (`departmentCode`, `position`) ถ้าปล่อยว่างไว้ ฟอร์มจะ error ด้วยข้อความ zod ดิบภาษาอังกฤษ ("Too small: expected string...") เพราะ react-hook-form ส่ง `""` ไม่ใช่ `undefined` — แก้ด้วย `z.preprocess` แปลง `""` เป็น `undefined` ก่อนเช็ค `.min(1)` (ต้องระวังลำดับ: `preprocess` ต้องห่อ schema ที่มี `.optional()` อยู่ข้างในแล้ว ไม่ใช่ห่อแล้วค่อย `.optional()` ข้างนอก มิเช่นนั้นจะพังคนละแบบ)
  2. หลังรัน `npx prisma generate` แล้ว dev server (ที่รันค้างอยู่ก่อนหน้า) ยังใช้ Prisma Client เวอร์ชันเก่าในหน่วยความจำ ทำให้ API error `Unknown argument titlePrefix` — **ต้อง restart dev server ทุกครั้งหลังแก้ schema + generate ใหม่** ไม่ใช่แค่รอ Fast Refresh เฉยๆ
  3. React เตือน "Select changing from uncontrolled to controlled" เพราะ default value เป็น `undefined` ตอน dropdown ยังไม่ถูกเลือก — แก้ด้วย `value={field.value ?? ""}` ที่ตัว `<Select>` ให้เริ่มต้นเป็น controlled ตั้งแต่ render แรกเสมอ
- ทดสอบสร้างผู้ใช้จริงผ่านฟอร์ม (นายสมชาย ใจดี, ฝ่ายไอที, หัวหน้าฝ่ายไอที) แล้วเปิดหน้าแก้ไขซ้ำเพื่อยืนยันว่าค่าที่บันทึกไว้โหลดกลับมาถูกต้องครบทุกฟิลด์ — ผ่าน

## 12. Module "สร้างเอกสาร" — เพิ่มประเภท "บันทึกข้อความ" เต็มรูปแบบ (2026-07-08 ค่ำ, ต่อจาก §11)

### 12.1 ขั้นตอน: วิเคราะห์ช่องว่างก่อน ไม่รีบแก้โค้ด
เทียบ `memo-form.html` + `memo-template-phitsanulok.md` (จาก session คู่ขนาน, §11.5) กับ schema/component ปัจจุบัน แล้วรายงานช่องว่างก่อนแก้จริงตามคำสั่งผู้ใช้ ("ยังไม่ต้องแก้โค้ดจนกว่าจะอนุมัติแผน") พบ 4 จุดต้องตัดสินใจร่วมกับผู้ใช้ก่อนเริ่ม:
1. **Format วันที่บันทึกข้อความ** ("พฤษภาคม 2569" ไม่มีวัน) → แยก format ตามประเภทเอกสาร (เก็บ `documentDate` เป็น `DateTime` เต็มเหมือนเดิมเสมอเพื่อความถูกต้องของข้อมูล/เรียงลำดับ แค่ชั้นการ**แสดงผล**ใน template เท่านั้นที่ต่างกัน)
2. **closingText** ("จึงเรียนมาเพื่อทราบฯ" / "ขอแสดงความนับถือ") → dropdown ตัวเลือกคงที่ ไม่ใช่ให้พิมพ์เอง
3. **บรรทัด "จาก {sender}"** ใต้ลายเซ็น → ซ่อนเฉพาะบันทึกข้อความ (ข้อมูลหน่วยงานย้ายไปอยู่ที่ "ส่วนราชการ" ในหัวเอกสารแทน)
4. **ข้อความอ้างอิงคำสั่ง/วันที่ที่มีจุดประ** ("...ที่.../2569 ลงวันที่........") → ฝังใน `content` แบบข้อความอิสระ ไม่แก้ schema เพิ่ม field ย่อย
