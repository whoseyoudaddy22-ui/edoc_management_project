## 5. ปัญหา Git ที่เจอและวิธีแก้ (สำคัญมาก — ใช้อ้างอิงครั้งต่อไป)

| ปัญหา | สาเหตุ | วิธีแก้ |
|---|---|---|
| Skill บางตัวไม่ปรากฏบนเครื่อง server หลัง pull | ไฟล์ถูก commit ไปอยู่บน feature branch (เช่น `feature/upgrade-ui`) ไม่ใช่ `master` | `git checkout <feature-branch> -- <path>` ดึงเฉพาะไฟล์มาที่ `master` แล้ว commit ใหม่ |
| `git checkout main` ขึ้น error `pathspec 'main' did not match` | repo นี้ใช้ชื่อ branch หลักว่า `master` ไม่ใช่ `main` | ใช้ `master` แทนทุกจุด |
| `git push` ขึ้น `[rejected] (fetch first)` | remote มี commit ที่ local ไม่มี (มีการ push จากอีกเครื่อง/อีกช่วงเวลา) | `git pull origin master` ก่อน แล้วค่อย push |
| `git pull` ขึ้น `divergent branches, need to specify how to reconcile` | local และ remote มีประวัติคนละชุดพร้อมกัน | `git config pull.rebase false` (เลือก merge ไม่ใช่ rebase) แล้ว `git pull` ใหม่ หรือใช้ `git pull --no-rebase origin master` ตรงๆ |
| commit ที่มีข้อความว่าใส่ 2 ไฟล์ แต่จริงมีแค่ 1 ไฟล์ | คำสั่ง `git checkout <branch> -- <path>` ก่อนหน้าล้มเหลวเงียบๆ (path ผิด) แต่ commit ทั้งสองคำสั่งไปในครั้งเดียว | ตรวจสอบด้วย `git log --oneline -- <path>` เทียบทีละไฟล์เสมอ อย่าเชื่อแค่ข้อความ commit |

**บทเรียนรวม:** เอกสาร/skill ควร commit แยกออกมาที่ `master` โดยตรงเสมอ ไม่ควรพ่วงไปกับ feature branch ที่กำลังพัฒนาโค้ดอยู่

## 6. Auth Bug ที่แก้ไปแล้ว

**อาการ:** หลัง login สำเร็จ กด Logout แล้ว redirect ไปที่ `http://localhost:3000` เกิด `ERR_CONNECTION_REFUSED` ทั้งที่เข้าเว็บผ่าน IP วง LAN (`192.168.1.152`)

**สาเหตุ:** Auth.js (next-auth v5) ตั้งค่า `AUTH_URL` แบบ hardcode เป็น `localhost:3000` ทำให้ระบบสร้าง redirect link ผิดเมื่อเข้าผ่าน IP อื่น

**วิธีแก้ที่เลือกใช้:** ตั้ง `AUTH_TRUST_HOST=true` แทนการ hardcode `AUTH_URL` — ให้ Auth.js อ่าน host จาก request จริงแทน (ยืดหยุ่นกว่า รองรับการเปลี่ยน IP/ใส่ Nginx ในอนาคตโดยไม่ต้องแก้ค่านี้ซ้ำ)

## 7. สถานะล่าสุดตอนจบบทสนทนานี้ (ตามผล `/status-check`)

- Module ครบ 16 ตามแผน + มี security hardening เพิ่มเติมหลัง Module 15 (password complexity, AuditLog immutability)
- Environment พร้อมสำหรับ dev/test แต่ **ยังไม่พร้อม go-live** เพราะขาด `.env.production` และ `.env.test`
- มี 5 commits ค้างอยู่บนเครื่อง server ที่ยังไม่ได้ push (รวม 2 security fix สำคัญ)
- `docker` group ของ session บนเครื่อง server ยังไม่ refresh (ต้อง logout/login)
- `docs/progress-log.md` ล้าสมัย ต้องอัปเดตให้ตรงกับ commit ล่าสุดจริง

### ขั้นตอนที่วางแผนไว้ต่อจากนี้ (ยังไม่ได้ทำ ณ จุดจบบทสนทนา)
1. Logout/login เพื่อ refresh docker group
2. Push 5 commits ค้างขึ้น origin
3. สร้าง `.env.production` และ `.env.test` (secret สุ่มใหม่ทั้งหมด ห้าม copy จาก `.env`)
4. รัน `prisma migrate deploy` กับ production database
5. อัปเดต `docs/progress-log.md`
6. รัน Testing (Module 14) + Security Review (skill `security-review`, แนะนำใช้ Opus)
7. ทดสอบ Backup/Restore จริงบนเครื่อง (สำคัญที่สุดเพราะไม่มี Snapshot)
8. Smoke test ผ่านหน้าเว็บจริงจาก LAN
9. ล้างข้อมูลทดสอบ + สร้างบัญชีผู้ใช้จริง
10. Tag เวอร์ชัน (`v1.0.0-golive`) และปิดจบ

## 8. แนวทาง Workflow ที่ตกลงกันไว้สำหรับงานถัดไป (UI Redesign)

- สร้าง `feature/ui-improvements` แยกจาก `master` เสมอ (`git checkout master && git pull && git checkout -b ...`)
- รันที่ port แยก (เช่น 3001) และ/หรือฐานข้อมูลแยก (`.env.test`) ระหว่างพัฒนา ไม่แตะ production ที่ user ใช้งานอยู่
- แก้ `ui-design-system/SKILL.md` ให้เป็นมาตรฐานใหม่ก่อน แล้วค่อยไล่แก้ทีละหน้า (Login → Sidebar/Layout → Dashboard → ฟอร์มสร้างเอกสาร → ตาราง/ค้นหา → อัปโหลด → คงหน้าพิมพ์เอกสารไว้ตามเดิมเพราะเป็นมาตรฐานราชการ → หน้าจัดการผู้ใช้/ประวัติ)
- ก่อน merge กลับ `master`: รัน test ทั้งหมด, security-review เฉพาะไฟล์ที่เปลี่ยน, smoke test มือ
- Deploy จริงต้อง tag ไว้ก่อนเสมอ (`git tag pre-ui-update-<วันที่>`) เพื่อมีจุด rollback
